# PlainChat 決定事項ログ

## 2026-07-04: 利用者スコープの確定
- **決定**: 本アプリは開発者本人の専用ツールとする(身内共有もしない)
- **理由**: 用途はChatGPT/Claude/Gemini等の商用チャットの代替であり、他者への提供は想定しない
- **影響**: ユーザー登録UIは作らない(初期セットアップ時にアカウント作成のみ)。
  Phase 4の「管理者メニュー」は不要方向で再検討。認証はJWT方式を踏襲するが、
  マルチユーザー管理機能は省略

## 2026-07-04: プロジェクト名の決定
- **決定**: プロジェクト名を「PlainChat」とする(リポジトリ名: plainchat)
- **理由**: ロールプレイ要素を持たない飾らない汎用会話ツール、というコンセプトを直接表す名前

## 2026-07-04: リポジトリ方針の決定
- **決定**: GitHub Public リポジトリとして作成(MIT License)
- **理由**: 将来的に勤務先など外部でも堂々と使える状態を目指すため
- **影響**: .env等の機密と実環境情報(サーバIP/ホスト名等)は一切コミットしない。
  環境固有の設定は .env.example のプレースホルダで表現する。
  実環境情報を含むドキュメントはプロジェクトナレッジ側のみで管理する

## 2026-07-04: NookResonanceとの関係の整理
- **決定**: NookResonanceの既存コードは初期段階の土台として流用してよい
  (認証等、実績のある実装のコピーを許可)
- **決定**: ただし実行時は完全独立とする。PlainChatはNookResonanceなしで単独動作し、
  別Node.jsプロセス・別ポート(18091)・別SQLite DBで運用する
- **影響**: 流用したコードはPlainChat側で自由に改変してよく、
  NookResonance側の変更に追従する義務は負わない

## 2026-07-04: LLM連携方式の確認と方針
- **発見**: NookResonanceの /api/llm/chat は非ストリーミング実装であり、
  「同一APIをそのまま流用」ではPhase 1要件(トークン逐次表示)を満たせない
- **決定**: PlainChatではSSEによるストリーミング中継を新規実装する。
  .env設定の読み込み・パラメータ組み立て・エラーハンドリングの構造は既存パターンを踏襲
- **備考**: 既存コードの [TEMP DEBUG] ログブロック等、一時的なコードは流用対象外

## 2026-07-04: 参照コードの配置方式
- **決定**: NookResonanceの全コードをローカルの .base/ に配置し、初期開発時の参照・流用元とする
- **決定**: .base/ は .gitignore 対象とし、リポジトリには絶対にコミットしない
- **備考**: 単独で安定動作するようになった段階で .base/ は削除してよい

## 2026-07-04: 実装プロンプトの管理方式
- **決定**: Claude Code向け実装プロンプトは prompts/queue/(未実行)と
  prompts/done/(実行済み)で管理する。ファイル名は連番+内容(例: 001_scaffold.md)
- **決定**: prompts/ はリポジトリにコミットする(機密を含まない設計のため、
  開発プロセスの記録として公開価値もある)

## 2026-07-05: 認証・認可の設計方針(利用者スコープ決定の補足)
- **決定**: MVPは本人専用のまま変えないが、認証/認可は正式に実装し、
  マルチユーザー対応可能な構造を最初から保つ
- **理由**: 将来的にローカルLLMを使いたい小規模顧客への提供可能性を残すため
- **影響**:
  - 全テーブルに user_id スコープを持たせ、全APIで所有者チェックを行う
  - users.is_admin カラムは保持する
  - ユーザー登録UI・ユーザー管理画面はMVPでは作らない(将来の追加余地として温存)
- **備考**: 2026-07-04「利用者スコープの確定」の「マルチユーザー管理機能は省略」を
  「管理UIは省略するが、データモデル・認可構造はマルチユーザー前提で設計」に修正
