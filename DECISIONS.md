# PlainChat 決定事項ログ

## 2026-07-04: 利用者スコープの確定
- **決定**: 本アプリは開発者本人の専用ツールとする(身内共有もしない)
- **理由**: 用途はChatGPT/Claude/Gemini等の商用チャットの代替であり、他者への提供は想定しない
- **影響**: ユーザー登録UIは作らない(初期セットアップ時にアカウント作成のみ)。
  Phase 4の「管理者メニュー」は不要方向で再検討。認証はJWT方式を踏襲するが、
  マルチユーザー管理機能は省略

## 2026-07-04: プロジェクト名の決定
- **決定**: プロジェクト名を「PlainChat」とする(リポジトリ名: plainchat)
- **理由**: ロールプレイ要素を持たない飾らない汎用会話ツール、というコンセプトを直接表す名前

## 2026-07-04: リポジトリ方針の決定
- **決定**: GitHub Public リポジトリとして作成(MIT License)
- **理由**: 将来的に勤務先など外部でも堂々と使える状態を目指すため
- **影響**: .env等の機密と実環境情報(サーバIP/ホスト名等)は一切コミットしない。
  環境固有の設定は .env.example のプレースホルダで表現する。
  実環境情報を含むドキュメントはプロジェクトナレッジ側のみで管理する

## 2026-07-04: NookResonanceとの関係の整理
- **決定**: NookResonanceの既存コードは初期段階の土台として流用してよい
  (認証等、実績のある実装のコピーを許可)
- **決定**: ただし実行時は完全独立とする。PlainChatはNookResonanceなしで単独動作し、
  別Node.jsプロセス・別ポート(18091)・別SQLite DBで運用する
- **影響**: 流用したコードはPlainChat側で自由に改変してよく、
  NookResonance側の変更に追従する義務は負わない

## 2026-07-04: LLM連携方式の確認と方針
- **発見**: NookResonanceの /api/llm/chat は非ストリーミング実装であり、
  「同一APIをそのまま流用」ではPhase 1要件(トークン逐次表示)を満たせない
- **決定**: PlainChatではSSEによるストリーミング中継を新規実装する。
  .env設定の読み込み・パラメータ組み立て・エラーハンドリングの構造は既存パターンを踏襲
- **備考**: 既存コードの [TEMP DEBUG] ログブロック等、一時的なコードは流用対象外

## 2026-07-04: 参照コードの配置方式
- **決定**: NookResonanceの全コードをローカルの .base/ に配置し、初期開発時の参照・流用元とする
- **決定**: .base/ は .gitignore 対象とし、リポジトリには絶対にコミットしない
- **備考**: 単独で安定動作するようになった段階で .base/ は削除してよい

## 2026-07-04: 実装プロンプトの管理方式
- **決定**: Claude Code向け実装プロンプトは prompts/queue/(未実行)と
  prompts/done/(実行済み)で管理する。ファイル名は連番+内容(例: 001_scaffold.md)
- **決定**: prompts/ はリポジトリにコミットする(機密を含まない設計のため、
  開発プロセスの記録として公開価値もある)

## 2026-07-05: 認証・認可の設計方針(利用者スコープ決定の補足)
- **決定**: MVPは本人専用のまま変えないが、認証/認可は正式に実装し、
  マルチユーザー対応可能な構造を最初から保つ
- **理由**: 将来的にローカルLLMを使いたい小規模顧客への提供可能性を残すため
- **影響**:
  - 全テーブルに user_id スコープを持たせ、全APIで所有者チェックを行う
  - users.is_admin カラムは保持する
  - ユーザー登録UI・ユーザー管理画面はMVPでは作らない(将来の追加余地として温存)
- **備考**: 2026-07-04「利用者スコープの確定」の「マルチユーザー管理機能は省略」を
  「管理UIは省略するが、データモデル・認可構造はマルチユーザー前提で設計」に修正

  ## 2026-07-05: Claude Codeの権限境界の確定
- **決定**: Claude Codeは git add / git commit を実行しない(ステージも不可)。
  コミットはレビュー後にユーザーが手動で行う
- **決定**: 指示ファイルの prompts/done/ への移動はClaude Codeの完了時作業とする。

- **理由**: 001と002で挙動が割れた(コミット有無、移動有無)。人間の関所をコミットに

- **備考**: 既存の実装プロンプト定型に共通ルール(第7項)として明文化

## 2026-07-05: 認可エラーの表現方針
- **決定**: リソースの所有者不一致は403ではなく404で返す(存在の推測を許さない)。
  「存在しない」と「他人の所有」をレスポンスで区別しない
- **理由**: 将来のマルチユーザー化時にリソースIDの存在が他ユーザーへ漏れるのを防ぐ
- **適用**: 004の会話CRUD以降、user_idスコープを持つ全APIで統一

## 2026-07-05: チャットエンドポイントとストリーム中断時の方針
- **決定**: メッセージ送信は POST /api/conversations/:id/chat に統一。
  ユーザー発話保存→履歴構築→SSE中継→アシスタント応答保存を1エンドポイントで行う
- **決定**: ストリーム途中の中断(クライアント切断/LLMエラー)時は部分応答を保存する。
  最初のトークン前の失敗時はアシスタントメッセージを保存しない
- **理由**: 途中まで受信した長い応答を失わせない。失敗時はユーザー発話のみ残し
  リトライ可能な状態を保つ

## 2026-07-05: LLMストリーミング中断時のkeep-alive問題への対処
- **発見**: SSE中継中にクライアント切断でAbortControllerを呼ぶと、Node(undici)の
  keep-aliveコネクションプールが壊れた状態で残り、以降そのプロセス内の
  全LLMリクエストが `terminated` エラーで失敗し続ける(サーバー再起動まで復旧しない)
- **決定**: llama.cpp serverへのリクエストヘッダーに `Connection: close` を付与し、
  リクエストごとにコネクションを使い捨てる
- **理由**: 1ユーザーのチャット中断が原因でサーバー全体のLLM接続が壊れる致命的な
  不具合のため。多少のTCPハンドシェイクのオーバーヘッドより安定性を優先

## 2026-07-05: フロントエンドUIの方針決定(最後の未決事項の解消)

## 2026-07-05: llama.cppクラッシュの根本原因判明
- **事実**: 生成途中クラッシュの正体は ggml-cuda/fattn.cu:579 の fatal error
  (CUDA Flash Attentionカーネル)。PlainChat/textgen-webuiは無関係
- **対処**: n_ctxを76800に戻し、max_tokens=32768として解消(インフラ側設定)
- **備考**: 以前の「切断伝播が原因」仮説は撤回。n_ctx=32000構成が
  かえって不安定だった。現行n_ctxは76800(当初計画の81920ではない点に注意)

## 2026-07-05: fattnクラッシュの解消確認
- **対処**: llama.cppを更新(修正ビルドへ差し替え)
- **確認**: 長い会話の継続を複数回実行し、クラッシュ再現なし。解決とみなす
- **残タスク**: MTP有効のままか確認、n_ctx(76800→81920)とLLM_MAX_TOKENS(4096→適正値)の
  最終値を決めて .env / インフラ設定を確定する

## 2026-07-05: NookResonance参照の終了

  (ローカル .base/ の削除と同時)。以後の一次参照はPlainChat自身のリポジトリ

## 2026-07-05: 空応答の防御方針
- **決定**: トリム後に空のassistant応答は保存せず、SSE errorとしてユーザーに通知する。
  履歴構築時も空contentメッセージをLLMリクエストから除外する
- **理由**: reasoningトークン超過による空応答がDBに残ると、以降の履歴を汚染し
  LLMサーバーの500を誘発する(008で実証)
- **関連**: 根本対策として .env の LLM_MAX_TOKENS 引き上げ(4096→8192以上)を推奨。
  chat本体への enable_thinking: false は導入しない(思考は応答品質に寄与)

  ## 2026-07-05: システムプロンプト設計
- **決定**: グローバル(users.system_prompt)+会話ごと上書き(conversations.system_prompt)の

- **決定**: systemメッセージはDBに保存せず送信時に動的挿入(設定変更は次の送信から反映)。

- **備考**: グローバルに長文(Claude系プロンプト改変版)を入れる場合、毎リクエストの
  プロンプト処理コストとモデルの指示遵守率に影響するため、運用しながら分量を調整する

- **決定**: {{currentDateTime}} / {{currentDate}} / {{currentTime}} を送信時に展開する
  (011)。DB保存値は変数のまま、LLM送信直前に置換。ユーザー発話には適用しない
- **決定**: グローバル設定にはClaude系プロンプト改変版を適用。knowledge cutoffは

## 2026-07-05: 会話の検索・フィルタの設計(012)
- **決定**: 検索は LIKE 部分一致方式(`title LIKE '%q%' OR 本文 content LIKE '%q%'`)。
  FTS5は採用しない。対象はタイトル+メッセージ本文の両方
- **決定**: 本文一致した会話には最小スニペット(最初にマッチしたメッセージの前後を
  切り出し、空白圧縮+省略記号)を1件だけ返す。タイトルのみ一致はスニペットnull
- **決定**: フィルタ軸は日付範囲(updated_at の from/to、両端含む)のみ。ロール等は持たない
- **決定**: 新エンドポイントは作らず、既存 GET /api/conversations を
  ?q= &from= &to= で拡張する。空クエリ時は現行どおり全件 updated_at 降順
- **決定**: ページネーションは本格実装せず LIMIT 100 の安全上限のみ設ける
  (004で本タスクに先送りされた宿題への結論)
- **理由**: 日本語コンテンツでは既定トークナイザのFTS5より LIKE 部分一致の方が
  素直に効く。単一ユーザー規模(数千メッセージ程度)では全走査でも実用上問題なく、
  スキーマ変更・同期トリガを避けられMVP規律に合致する
- **ロードマップ**: FTS5(trigramトークナイザ+snippet())はデータ増大時の
  Phase 3+候補として記録。真のページネーションも同様に先送り

## 2026-07-05: 画像添付まわりの設計判断
- **配信/表示認証**: 画像はattachment idで配信(GET /api/uploads/image/:id、Bearer必須)。
  フロントは認証付きfetch→Blob URLで表示し、URLクエリにJWTを載せない(Public リポジトリでの
  秘密露出を避けるため。NookのURLトークン方式は採らない)
- **保存先**: data/uploads/{userId}/ に衝突しにくいファイル名で保存。会話・メッセージとの
  紐付けはattachmentsテーブルのカラムで持ち、ファイルパスには含めない(パスに会話IDを
  埋めないことでmulterのフィールド順依存を回避)
- **履歴再送**: Visionのフォローアップ成立のため、画像を含むメッセージは後続ターンでも
  画像を再送する。CTX 81920前提で当面許容。将来「Nターン超で古い画像を落とす」最適化を候補記録
- **attachmentsスキーマ**: 独立テーブル(id, user_id, conversation_id, message_id NULL可,
  kind, mime, size, path, original_name, created_at)。user_idスコープ・不一致404。
  ②PDF添付も同テーブルを再利用する

  ## 2026-07-05: 画像添付は複数対応(上限つき・フロント方針)
- **決定**: フロントの画像添付は複数対応とする。1メッセージあたりの上限は4枚(フロントの定数)。
  013/014のAPIは既に配列対応のためバックエンドは無改造。上限もフロント側のみで制御する
- **理由**: 複数枚の比較・参照が実用的。画像はトークンが重く履歴再送方針のため、
  暴発防止に上限を設ける。単一ユーザー運用ゆえバックエンド側の上限は当面設けない
- **未検証事項**: textgen-webui経由での「1メッセージ複数画像」の素通しは015の実地確認で担保する
  (014の実地確認は1枚)。ドラッグ&ドロップ・貼り付けは将来候補

## 2026-07-05: 空応答対応(reasoning budgetで根本緩和)
- **決定**: 空応答(思考がmax_tokensを食い尽くしcontentが空)への対策として、
  llama.cpp serverに reasoning budget を設定する。textgen-webuiがllama.cppを内部起動
  しているため、ローダの extra-flags に `--reasoning-budget N`(初期6000)と
  `--reasoning-budget-message` を渡す。全リクエスト一律の上限とする
- **理由**: 思考と答えが同一のmax_tokens枠を奪い合うのが根本原因。budgetで思考を打ち切り
  答えの枠を必ず残す。インフラ設定のみでPlainChat無改造、2段構成でも確実に効く
- **割り切り**: グローバルフラグ設定中はリクエスト単位の thinking_budget_tokens 上書きは
  無効(タスク別制御が要るなら将来ボディ方式+textgen素通し確認へ)
- **据え置き**: 009の空応答防御は最終防壁として維持。reasoning_content を空時に救済する
  chat.js硬化(④)は必要時の候補
- **要確認**: Gemma構成で思考タグがbudget機構に認識されるか(reasoning解析がautoで効くか)を
  実地確認。効かなければ reasoning on 明示/テンプレートのthinkタグを点検

  ## 2026-07-05: 空応答対応(reasoning budgetで解決)
- **決定**: 空応答(思考がmax_tokensを食い尽くしcontentが空)への対策として、
  llama.cpp serverに reasoning budget を設定。textgen-webuiがllama.cppを内部起動しているため、
  ローダの extra-flags に `--reasoning-budget 6000` を渡す(全リクエスト一律)
- **理由**: 思考と答えが同一のmax_tokens枠を奪い合うのが根本原因。budgetで思考を打ち切り
  答えの枠を必ず残す。インフラ設定のみでPlainChat無改造、2段構成でも確実に効く
- **検証**: 長文グローバルSystem Prompt(約11,600字)をそのまま使用し、テキスト/合成画像2枚/
  実写2枚の全パターンで空応答が再現しないことを確認(応答4〜9秒)。フロント無改造
- **割り切り**: グローバルフラグ設定中はリクエスト単位の thinking_budget_tokens 上書きは無効
  (タスク別制御が要るなら将来ボディ方式+textgen素通し確認へ)
- **据え置き**: 009の空応答防御は最終防壁として維持。reasoning_content を空時に救済する
  chat.js硬化(④)は必要時の候補
- **運用注意**: extra-flags 経由でスペースを含む値(--reasoning-budget-message 等)を渡すと
  llama-server起動時に引数が途中で切れて落ちる。まず --reasoning-budget 単体で運用し、
  メッセージ注入が要るなら値を1語にするかクォートで囲む
- **補足**: 移行メッセージも併用。extra-flags は
  `--reasoning-budget 6000 --reasoning-budget-message "Thinking budget exceeded, let's answer now."`。
  スペースを含む値はダブルクォートで囲めば textgen 経由でも1引数として渡る

## 2026-07-05: Phase 3 ② テキストファイル添付のスコープ確定
- **決定**: ②はテキスト系ファイル(txt / Markdown / csv / json、UTF-8)の添付・要約・QAに絞る。
  PDF・OCR・docx・pptx は今回見送り(後続の独立レバー)。csv/json は整形せず生テキストで注入
- **注入**: 添付本文は履歴に残る限り毎ターン再注入する(画像の再送方針をファイルにも適用)。
  ③「圧縮に頼らない資料相談」はこの注入方式に吸収し、独立タスクにはしない
**上限(補足)**: 「4万文字」は1ファイルあたりの注入上限。複数ファイル×履歴再注入での
  累積が81920を圧迫し得るのは許容し、恒常的に溢れる運用が出たらPhase 4のトークン予算管理で対応
- **複数/混在**: テキストファイルは1メッセージ最大4ファイル。画像(最大4)との混在を許可し、
  枠は別々にカウントする(画像4 + ファイル4、独立)

## 2026-07-07: 028 スキーマと封筒暗号の確定
- **決定(スキーマ mcp_servers)**: label/enabled/transport/command/args/url/catalog_id/sort_order は平文。
  秘密は env と headers を「別カラムの独立暗号文」で持つ: env_enc/env_iv/env_tag、headers_enc/headers_iv/headers_tag。
  片方のみ保持(NULL可)を許容。command/args/url は平文(秘密ではなく起動方法。command/args は029のカタログ方式で書込を縛る)
- **決定(封筒暗号)**: AES-256-GCM(Node標準 crypto、追加依存なし)。鍵は .env の SECRET_ENC_KEY(base64の32バイト)のみ、
  DBに鍵を置かない。IVは毎回ランダム、authTag保存。復号は使用直前のみ、復号値はログ/レスポンス/エラーに出さない
- **決定(退避)**: SECRET_ENC_KEY 未設定/不正時、暗号項目を持つサーバーはスキップしPlainChat起動は継続。
  暗号項目を持たないサーバーは鍵無しでも起動可
- **決定(loadMcpServers)**: 内部のみ DB読込+復号へ差し替え、返り値形 {label,enabled,transport,command,args,env(,url,headers)} は027から不変
- **決定(自動seed)**: 028a は初回起動時に mcp_servers が空かつ DEBUG_searxng 設定済みなら searxng を暗号化seed(後方互換)。
  この後方互換seedと DEBUG_searxng 依存、builtin 一式は 030 で撤去し「DBが唯一の源」の形へ寄せる
- **タスク**: 028a=テーブル+secretBox+loadMcpServers差替+鍵未設定退避+自動seed(UIなし土台)/
  029=設定画面(管理者ゲート・カタログ方式)/ 030=builtin撤去+後方互換seed/DEBUG_searxng撤去

## 2026-07-07: タスク順を再入れ替え(設定UIを先行) — 秘密投入の正規経路を最優先
- **決定(順序)**: 029=設定UI+格納(接続なし)、030=HTTPトランスポート対応(実接続)、031=builtin撤去。
  旧「029=HTTP接続 → 030=UI」を逆転する
- **理由**: HTTP型MCP(例 別プロジェクト)の認証はトークンを headers に持つ。トークン投入の正規経路(UI)が
  無いまま接続を先に作ると、手SQLでの暗号文直挿しやチャット/CLIへのトークン露出、Claude Code へのトークン
