# 041: フロント資産キャッシュの自動バージョニング

## 1. 目的
css/style.css・js/api.js・js/app.js への手書き?v=NNNクエリ運用を廃止し、ファイル内容のハッシュを
サーバー側で自動計算してindex.html配信時に注入する仕組みに置き換える。フロント改修時のバージョン
更新忘れによるブラウザキャッシュ事故(035で発見)を構造的に無くす。

## 2. 対象
Phase 4「改善フェーズ」。DECISIONS.md「2026-07(041): フロント資産キャッシュの自動バージョニング」
に従う。

## 3. 前提・参照
- 現状、public/index.html に js/api.js?v=036・js/app.js?v=039 のように手書きの ?v=NNN が
  埋め込まれている。フロント改修のたびに該当箇所を手で書き換える運用だった(035/037/038/039/
  039-fix/040でこの運用が繰り返された)
- public/ 配下は素のHTML/CSS/JS(ビルドツールなし、DECISIONS.md 2026-07-05参照)。この方針は
  変えない。今回もNode側のexpress.staticとルーティングだけで完結させる
- index.htmlは現状おそらくexpress.staticの既定index配信(GET /やGET /index.htmlで自動的に
  返される)に乗っている。これを専用ルートに切り替える必要がある
- src/index.js に既存のExpressエントリポイント・ミドルウェア登録がある(認証・静的配信・
  各routes/*のマウント等)。今回追加するindex.html用ルートは、既存の認証保護されたAPIルートとは
  別枠(index.html自体は未ログインでも返してよい。ログイン画面もこの1ファイルの中でJS制御されている)
- prompts/ 配下は指示書。移動以外の編集はしない

## 4. 要件

### (a) アセットバージョン計算ヘルパ(新規: src/assetVersion.js 等)
- getAssetVersion(absoluteFilePath) のような関数を用意する:
  - 呼び出しごとに fs.statSync でmtimeMsを取得(軽量)
  - モジュール内のメモリキャッシュ(Map等)に { mtimeMs, hash } を保持し、
    前回計算時のmtimeMsと一致すれば再計算せずキャッシュ済みhashを返す
  - mtimeMsが変化していれば、ファイル内容を読み込みsha256を計算し、16進文字列の先頭8〜10文字程度に
    切り詰めた短いバージョン文字列を返す(キャッシュも更新する)
  - 対象ファイルが存在しない・読み込み失敗時は例外を投げず、警告ログを出し固定のフォールバック文字列
    (例 'na')を返す(1ファイルの異常でページ全体が500にならないようにする)

### (b) index.html配信の専用ルート化(src/index.js または新規routeファイル)
- express.static の index.html 自動配信を無効化する(例: express.static(publicDir, { index: false })。
  他の静的ファイル(css/js/画像/manifest等)の配信は従来通りexpress.staticに任せる
- GET / と GET /index.html(両方)に対応するハンドラを追加し、以下を行う:
  - public/index.html を都度ファイルから読み込む(起動時1回だけキャッシュせず、リクエストごとに
    読み込む。理由: 開発中にindex.html自体を編集した場合も再起動なしで反映されるようにするため。
    ファイルサイズは小さいため性能上の懸念はない)
  - html内の css/style.css・js/api.js・js/app.js への参照(既存の?v=NNNが付いたものを含む)を、
    (a)のヘルパで計算した現在のバージョン文字列を使った ?v=<hash> 付きURLに置き換える
    (既存の手書き?v=NNNは完全に不要になるため、public/index.htmlのソース自体からも
    ?v=NNNの記述を削除し、素の src="js/app.js" 等に戻しておく。置換ロジック側は
    「?v=...が付いていてもいなくても正しく現在のハッシュへ揃える」形にしておくと安全)
  - レスポンスヘッダに Cache-Control: no-cache を設定してから返す(text/htmlはExpressのres.sendで
    文字列を渡せば自動設定される想定。明示してよい)

### (c) public/index.html のソース更新
- js/api.js・js/app.js・css/style.cssへの参照から手書きの ?v=NNN を削除し、
  素のパス(例 src="js/app.js")に戻す(バージョンは(b)のルートが動的に付与するため)

## 5. やらないこと
- 静的資産(css/js自体)へのCache-Control長期キャッシュ(max-age等)の追加設定。今回はindex.html側の
  no-cacheとバージョン付きURLの自動化のみ
- ビルドツール・バンドラの導入(素のHTML/CSS/JS方針は維持)
- manifest.json・アイコン類へのバージョニング適用
- Service Workerの導入(034の決定どおり導入しない)
- 025/032/039/040等、既存のバックエンドAPI・MCP・チャットロジックの変更
- .envの変更

## 6. 完了条件
1. サーバー起動後、ブラウザで http://localhost:18091/ を開くと、開発者ツールのNetworkタブで
   js/app.js・js/api.js・css/style.css がそれぞれ ?v=<8〜10桁の16進文字列> 付きで読み込まれている
   ことを確認する
2. index.htmlのレスポンスヘッダに Cache-Control: no-cache が付いていることを確認する
3. public/js/app.js の中身を1文字でも変更して保存 → サーバーを再起動せずにブラウザをリロード →
   app.jsのバージョン文字列(ハッシュ)が変わり、かつ実際に新しい内容が反映されていることを確認する
   (これがmtimeベースキャッシュ無効化の動作確認)
4. 何も変更せずに複数回リロードしても、同じバージョン文字列が返り続ける(毎回ハッシュ再計算されて
   いないことをレスポンス内容の一致で間接確認、またはログ等で確認できれば尚可)
5. public/index.html のソースに ?v=NNN の手書き記述が残っていないことを確認する
6. css/style.css を変更した場合も同様にバージョン文字列が変わり反映されることを確認する
7. 対象外の静的ファイル(画像・manifest.json等)が従来通り正常に配信されることを確認する
   (express.staticの{index:false}化が他の配信に影響していないこと)
8. ログイン画面・チャット画面など、index.html配信に依存する既存の画面遷移が壊れていないことを
   ブラウザで一通り確認する
9. 検証用に変更したファイルの内容を元に戻す必要があれば戻し、完了する
