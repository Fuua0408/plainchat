# 012: 会話の検索・フィルタ

## 1. 目的
会話をタイトルおよびメッセージ本文で検索し、日付範囲で絞り込めるようにする。
サイドバーから目的の会話に素早く到達できる状態を作る(Phase 2の最後の項目)。

## 2. 対象
Phase 2「会話の検索・フィルタ」

## 3. 前提・参照
- 001〜011完了済み。src/routes/conversations.js の GET /api/conversations
  (自分の会話を updated_at 降順で全件返す)を拡張する
- 設計方針(DECISIONS.md 2026-07-05「会話の検索・フィルタの設計(012)」):
  - LIKE 部分一致方式。FTS5・スキーマ変更は行わない
  - 検索対象はタイトル+メッセージ本文の両方
  - 既存エンドポイントを拡張(新エンドポイントは作らない)
  - 認可は既存方針を踏襲(user_idスコープ、所有者不一致・非存在は404)
- prompts/ 配下は指示書であり、移動以外の編集はしない

## 4. 要件

### バックエンド: GET /api/conversations の拡張(src/routes/conversations.js)
クエリパラメータ(すべて任意):
- q    : 検索文字列。前後空白をトリム。トリム後に空なら「指定なし」とみなす
- from : 日付下限(YYYY-MM-DD)。updated_at がこの日以降(当日含む)
- to   : 日付上限(YYYY-MM-DD)。updated_at がこの日まで(当日含む)

挙動:
- q も from も to も無い場合 → 現行どおり自分の会話を updated_at 降順で返す
  (後方互換。既存フロントの呼び出しを壊さない)
- q がある場合の一致条件(OR):
  - conversations.title が q を部分一致で含む
  - その会話のいずれかの messages.content が q を部分一致で含む
  - 同一会話は重複させない(DISTINCT 相当)
- 日付フィルタは date(updated_at) を基準に from/to と比較(両端含む)。
  q の有無に関わらず併用可能(q と AND で結合)
- 並び順は updated_at 降順を維持
- 上限 LIMIT 100(本格ページネーションは実装しない)

LIKE の安全な取り扱い:
- q 内の LIKE ワイルドカード(% _ および エスケープ文字)をエスケープし、
  ESCAPE 句付きで比較する。ユーザーが「%」等をリテラルとして検索できること
- LIKE の大小文字非区別(ASCII)は既定のままでよい

スニペット(本文一致時のみ):
- q があり、かつタイトルではなく本文で一致した会話について、
  最初にマッチしたメッセージ(id 昇順で先頭)からスニペットを生成して返す
- スニペット生成: 一致位置の前後を切り出した約80〜100文字の窓。
  改行・連続空白は単一スペースに圧縮。先頭/末尾を切った場合は … を付す
- タイトルのみ一致・q 指定なしの場合は snippet は null(または省略)

バリデーション:
- q はトリム後の長さ上限200文字。超過は400
- from / to が YYYY-MM-DD として不正な形式なら400
- 認証・所有者スコープは既存 GET /api/conversations と同一(トークンなし401、
  他ユーザーの会話・メッセージは一切ヒットしない)

レスポンス形:
- 既存形 { conversations: [{ id, title, created_at, updated_at }] } を維持しつつ、
  各要素に snippet(string | null)を追加する
- フロントは snippet の有無どちらも扱えるようにする

性能について:
- 本文の LIKE '%q%' はインデックスを使えず全走査になるが、単一ユーザー規模では
  許容する(DECISIONS.md 参照)。ここで最適化(FTS5等)は行わない

### フロントエンド(public/ 配下)
- サイドバー上部(会話一覧の上)に検索ボックスを追加
  - 入力を250〜300msデバウンスして GET /api/conversations?q=... を呼び、一覧を差し替え
  - クリアボタン(×)で検索文字列を消し、全件表示に戻す
- 日付範囲フィルタ
  - 検索ボックス付近に、折りたたみ可能な範囲で from / to の <input type="date"> を2つ置く
    (常時展開でクラッタになるなら小さなトグルで開閉。実装しやすい方でよい)
  - 変更時に現在の q と合わせてAPIを呼び直す
- 一覧項目の表示
  - 従来のタイトル+削除ボタンに加え、snippet があれば控えめな補助テキストとして
    タイトル下に1行表示(はみ出しは省略)
  - 既存の挙動(クリックで開く、現在の会話をハイライト、削除)は維持
- 該当0件のとき「該当する会話がありません」等の空表示を出す(一覧が消えて固まらない)
- 「新しい会話」作成時は検索文字列・日付フィルタをクリアして全件表示に戻してから
  新規会話を開く(新規会話が検索条件に埋もれて見えない事故を防ぐ)
- 送信完了・タイトル自動生成・リネーム後の一覧更新は、現在アクティブな検索条件を
  保ったまま再取得してよい(並び順が変わるのは許容)

## 5. やらないこと

- 検索語のハイライト装飾以上の凝ったスニペット表示、複数スニペット
- 検索履歴・保存済み検索・タグ等の新概念
- chat.js / db.js / auth.js / 認証まわりの変更(検索は conversations.js で完結)
- .env の変更

## 6. 完了条件
curl(Bearer付き)およびブラウザで確認:
1. タイトルに含まれる語で検索 → 該当会話が返る
2. タイトルには無いが本文に含まれる語で検索 → その会話が返り、snippet が付く
3. 本文一致会話の snippet に一致周辺のテキストが入り、改行が潰れている
4. 検索語に % や _ を含めても、リテラルとして扱われ誤爆しない
5. from / to での日付範囲絞り込みが両端含みで効く(q併用・q無し両方)
6. q も from も to も無いとき → 現行と同じ全件 updated_at 降順(既存フロント無改修でも動く形)
7. 101件以上該当する状況を作れる場合、返却が100件で頭打ちになる
   (作れなければクエリ上の LIMIT 100 をコードで確認)
8. q が201文字 → 400、from/to に不正な日付 → 400、トークンなし → 401
9. DBに別ユーザーの会話・メッセージを一時作成し、その語で検索しても
   自分の結果に一切現れないこと → 確認後にテストデータ削除
10. ブラウザ: 検索ボックスでの絞り込み、クリアでの復帰、日付フィルタ、
    該当0件表示、新規会話作成で検索がクリアされることを確認
11. コンソールに未処理エラーがない
12. テストで作成した会話・メッセージを削除して完了
