# 009: 既存バグ修正(リネームUI・空応答の防御)

## 1. 目的
008で発見された2つの問題を修正する:
(a) タイトルリネームのダブルクリックが機能しないUIバグ、
(b) 空のassistant応答がDBに保存され履歴を汚染する問題。

## 2. 対象
Phase 2(品質改善)。新機能なし

## 3. 前提・参照
- 008完了済み。バグの原因分析は008の実行報告どおり:
  (a) タイトルspanのclickハンドラがselectConversation→renderConversationListで
  一覧DOMを再生成するため、dblclickが切り離された要素に発火する
  (b) モデルがreasoningにトークンを使い切るとcontentが空のまま完了し、
  空文字列のassistantメッセージが保存される。空メッセージを含む履歴は
  LLMサーバーの500を誘発することも確認済み
- prompts/ 配下は指示書であり、移動以外の編集はしない

## 4. 要件

### (a) リネームUIの修正(public/js/app.js)
- ダブルクリックでリネーム入力欄が確実に表示されるようにする
- 修正方針は任意(例: click側をシングル/ダブル判定で遅延させる、
  再レンダリングを避けて選択ハイライトのみ更新する、等)。
  採用した方式と理由を報告すること
- シングルクリックでの会話切り替えの体感を悪化させない(顕著な遅延を入れない)

### (b) 空応答の防御(src/routes/chat.js)
- ストリーム完了時、fullTextがトリム後に空なら:
  - assistantメッセージを保存しない
  - event: error / data: {"error":"LLMの応答が空でした(思考トークン超過の可能性)"}
    を送出して終了(フロントは既存のonErrorで表示される)
- 履歴構築時、contentがトリム後に空のメッセージをLLMへ送る配列から除外する
  (DB上の既存の空メッセージにも耐性を持たせる)
- 部分応答保存(中断時)の既存挙動は変えない(空でない部分応答は従来どおり保存)

## 5. やらないこと
- enable_thinking の chat.js への導入(通常会話の思考は品質に寄与するため維持)
- リネームUIのデザイン変更、その他のUI改修
- DB内の既存の空メッセージの一括削除(履歴構築側の除外で無害化されるため)
- .env の変更

## 6. 完了条件
1. ブラウザ(または実DOMイベント)でダブルクリック→リネーム入力欄が表示され、
   確定でタイトルが変わる。シングルクリックの会話切り替えも正常
2. 空応答をシミュレート(max_tokensを極小にする等、手段は任意)し、
   assistantメッセージが保存されず、フロントにエラーが表示されることを確認
3. DBに空contentのassistantメッセージを直接作成した会話で送信し、
   LLMへのリクエストに空メッセージが含まれず正常応答することを確認
   → テストデータは削除
4. 通常の送受信・中断時の部分応答保存が従来どおり動くこと(リグレッション確認)
5. テストで作成した会話は削除して完了
