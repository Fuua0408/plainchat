# 023: 添付ファイルの削除時クリーンアップ+孤児回収

## 1. 目的
会話削除時に、その会話の添付実ファイルを削除する。あわせて起動時に、DBが参照していない
アップロード実ファイル(過去のテスト残骸を含む)を回収する。ディスクにゴミを残さず
Phase 3 ②を締めるための後片付け。

## 2. 対象
Phase 3 ②「ファイル添付」の締め。013以降「別タスク」として先送りしてきた
削除時クリーンアップの回収(DECISIONS.md 2026-07-05「Phase 3 ② 完成」の残タスク)。

## 3. 前提・参照
- 013〜022完了済み。attachments テーブルは (id, user_id, conversation_id, message_id,
  kind, mime, size, path, original_name, created_at)。実ファイルは data/uploads/{userId}/ 配下
- src/attachmentStorage.js が DATA_DIR の絶対パス化と
  resolveAttachmentFilePath(attachment) を提供(013〜017共有)
- 会話削除は src/routes/conversations.js の DELETE /api/conversations/:id。
  messages は ON DELETE CASCADE で消える(attachments行がFKでcascadeするかは要確認。
  cascadeしないなら本タスクで明示削除する)
- 認可は user_idスコープ、所有不一致は404(既存方針)
- prompts/ 配下は指示書であり、移動以外の編集はしない

## 4. 要件

### (a) 会話削除時の実ファイル削除(conversations.js)
- DELETE /api/conversations/:id で会話を削除する際、その会話に紐づく attachments の
  実ファイルを削除する
- 手順: 会話の所有者チェック(既存)→ その conversation_id の attachments を取得 →
  各実ファイルを resolveAttachmentFilePath で解決して unlink → 会話(と関連行)を削除
- attachments 行が messages/conversations の CASCADE で消えるか確認し、消えないなら
  明示的に DELETE する(実ファイル削除の後、行も残さない)
- ファイルが既に無い場合(欠損)や unlink 失敗は、致命エラーにせずログに残して続行する
  (会話削除API自体は成功させる)
- 削除後、その会話の空になったユーザーディレクトリ配下の残骸は無理に消さなくてよい
  (ディレクトリ削除は孤児回収側に委ねる。ここではファイル削除を確実に行う)

### (b) 起動時の孤児ファイル回収(src/db.js 初期化後 または起動処理内)
- サーバー起動時に一度、data/uploads/ 配下を走査し、attachments テーブルの path 集合に
  含まれない実ファイル(=DBが一切参照していないファイル)を削除する
- 安全策(誤削除防止、必須):
  - 走査対象は DATA_DIR 配下の uploads ディレクトリに厳密に限定する
    (絶対パスに正規化し、uploads ルートの外を絶対に触らない)
  - attachments の全 path を解決した絶対パス集合を作り、それに一致しない「ファイル」のみ削除。
    判断がつかないもの(想定外の形状・シンボリックリンク等)は削除せず残す
  - 削除したファイルは件数とパスをログに出す(何を消したか追跡できるように)
- 起動時処理は失敗してもサーバー起動を止めない(try/catchで隔離し、ログに残して続行)
- 空になったユーザーディレクトリは削除してよい(任意。実装したら報告)

## 5. やらないこと
- attachments スキーマの変更
- アップロード/配信API(013/016)・chat.js・注入ロジックの変更
- 定期実行・cron 的な仕組み(起動時1回の回収で十分。常駐ジョブは作らない)
- 個別メッセージ単位の削除API(現状メッセージ単体削除UIは無い。会話削除のみ対象)
- ゴミ箱/論理削除・復元機能(物理削除でよい)
- uploads ルート外への一切のファイル操作
- .env の変更

## 6. 完了条件
実DB・実サーバーで確認:
1. 添付(画像+ファイル)を含む会話を作成 → data/uploads/{userId}/ に実ファイルが存在 →
   その会話を DELETE → 実ファイルが消え、attachments 行も残らない
2. 会話削除APIは、実ファイルが既に欠損していても 200 で成功する(ログに欠損を記録)
3. DBに存在しない実ファイルを data/uploads/{userId}/ に手動で置く → サーバー再起動 →
   その孤児ファイルが削除され、ログに削除件数/パスが出る
4. DBが参照している実ファイル(生きている添付)は、再起動後も削除されないことを確認
   (孤児回収が生きたファイルを誤って消さない)
5. uploads ルート外(例: data/ 直下や data/uploads の外)にファイルを置いても、
   それらには一切触れないことを確認
6. 起動時の回収処理でエラーが出てもサーバーが正常に起動する(処理を握りつぶして続行)
7. 過去のテスト残骸がある場合、再起動後に data/uploads/ からそれらが一掃されている
   (生きている添付は残る)
8. テストで作成した会話・添付を削除して完了
